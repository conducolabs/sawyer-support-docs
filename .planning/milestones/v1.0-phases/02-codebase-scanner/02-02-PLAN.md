**Step 2: Create src/scanner/prompts.ts** with prompt constants:

- `SYSTEM_PROMPT`: Base system prompt for all passes. Instructs Claude to analyze a codebase for user-facing features. Specifies the exclusion rules: ignore navigation/routing structures, loading/error states, auth guards/wrappers, dev/debug screens. Specifies that features can be single screens OR multi-screen flows — Claude uses judgment.

- `PASS1_PROMPT`: "Identify all user-facing screens and routes in this codebase. For each, report the file path, component name, and a brief description of what the screen does. Focus on screens that end users interact with. Report any navigation library you detect (React Navigation, Expo Router, Next.js pages/app router, etc.)."

- `PASS2_PROMPT(pass1Result: string)`: Takes Pass 1 output. "Given these identified screens, classify each as either 'user_facing' or 'infrastructure'. User-facing: screens where users accomplish tasks (view data, submit forms, navigate features). Infrastructure: loading screens, error boundaries, navigation wrappers, auth guards, dev tools, splash screens. Return only user-facing screens."

- `PASS3_PROMPT(pass2Result: string, repoType: 'mobile' | 'dashboard')`: Takes Pass 2 output. For mobile: audience is end_user. For dashboard: classify admin role levels (club_admin, company_admin, super_admin) per feature. "For each user-facing screen, extract: canonical feature name (use the most stable/official name — e.g., 'Check-in' not 'Checkin Screen'), feature area grouping (e.g., 'authentication', 'payments', 'check-in'), and a 1-2 sentence description of what the feature does from the user's perspective."

- `PLATFORM_PROMPT`: "Analyze this API codebase. For each endpoint group, extract: the resource name, HTTP methods available, the data model fields, and any business logic relevant to end-user documentation. Do NOT classify these as features — they provide context for features identified in the mobile and dashboard scans."

**Step 3: Create src/scanner/passes.ts:**

- Import `query` from `@anthropic-ai/claude-agent-sdk`
- Import prompts from `./prompts.js`
- Import `buildSlug` from `../paths/index.js`
- Import `FeatureMapSchema`, `Feature` from `./schema.js`

- `runDiscoveryPass(repoPath: string, logger: Logger): Promise<string>` — Pass 1. Uses `query()` with `cwd: repoPath`, `allowedTools: ['Read', 'Grep', 'Glob']`, `permissionMode: 'bypassPermissions'`, `allowDangerouslySkipPermissions: true`, `maxTurns: 15`. Streams partial messages via logger.info. Returns the result text.

- `runClassificationPass(repoPath: string, pass1Result: string, logger: Logger): Promise<string>` — Pass 2. Same options, uses PASS2_PROMPT(pass1Result). Can use `resume` session if available. Returns filtered result text.

- `runExtractionPass(repoPath: string, pass2Result: string, repoType: 'mobile' | 'dashboard', logger: Logger): Promise<Feature[]>` — Pass 3. Uses `outputFormat: { type: 'json_schema', schema: featureJsonSchema }` where featureJsonSchema is a hand-written JSON Schema object matching the Feature type (do NOT use Zod toJsonSchema — hand-write per RESEARCH.md open question). Extracts structured_output, applies buildSlug to each feature name for the slug field, sets sourceApp based on repoType. Validates each feature with FeatureSchema.safeParse.

- `runPlatformPass(repoPath: string, logger: Logger): Promise<string>` — Uses PLATFORM_PROMPT, returns API context as text string. This context gets attached to related features in the merge step.

All pass functions MUST set both `permissionMode: 'bypassPermissions'` AND `allowDangerouslySkipPermissions: true` (per RESEARCH.md pitfall 2). All pass functions MUST use absolute paths for `cwd` — resolve repoPath with `path.resolve()`.

**Step 4: Create src/scanner/merge.ts:**

- `mergeFeatureMaps(mobileFeatures: Feature[], dashboardFeatures: Feature[], platformContext: string): FeatureMap` — Combines mobile and dashboard features into a single FeatureMap. For features found in both (matched by slug), sets sourceApp to 'both'. Attaches relevant platformContext to features as apiContext string. Sets generatedAt to current ISO timestamp.

**Step 5: Update src/scanner/index.ts** to export from passes.ts and merge.ts.
