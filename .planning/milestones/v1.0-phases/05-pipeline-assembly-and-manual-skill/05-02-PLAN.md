    For `buildFeatureFromArgs` tests: verify it creates a Feature-shaped object with `buildSlug(name)` as slug, provided audience/sourceApp/featureArea, empty screens array, and empty description (skill-created articles don't have scanner descriptions).

    Then implement `src/skill/new-article.ts` as a standalone script with three subcommands parsed from `process.argv`:

    **`check-slug <slug> --area <area>`**
    - Calls `checkSlugCollision(cwd, area, slug)`
    - Prints JSON: `{ "exists": true/false, "path": "docs/de/{area}/{slug}.md" }`
    - Exit 0

    **`generate --feature <name> --app <app> --audience <audience> --area <area> --type <type>`**
    - Calls `buildFeatureFromArgs()` to construct a Feature object
    - Calls `runGeneration(feature, config.model)` to get content
    - Prints the generated content to stdout (SKILL.md will display it to developer)
    - Exit 0

    **`write-and-translate --slug <slug> --area <area>`**
    - Reads stdin for the article content (German draft)
    - Calls `writeArticle(cwd, feature, content)` to write German .md
    - Creates DeepL client, translates to all non-de config languages
    - For each language: `parseFrontmatter` the German content, `computeHash` it, `translateArticle`, `buildTranslatedFrontmatter`, `writeTranslatedArticle`
    - Prints summary of files written
    - Exit 0

    Export the helper functions (`checkSlugCollision`, `buildFeatureFromArgs`, `writeAndTranslate`) for testability. Guard the CLI entry point with:

    **Body (markdown instructions for Claude):**

    The body must instruct Claude to follow this exact flow:

    **Step 1: Gather Information**
    Ask these questions one at a time (wait for each answer before the next). If `$ARGUMENTS` is provided, use it as the answer to question 1 and skip asking it:
    1. What feature or topic should this article cover? (If answer is vague or unclear, ask clarifying questions before proceeding -- e.g., "Can you be more specific about which part of [topic]?" or "Is this about the mobile app feature or the dashboard admin feature?")
    2. Which app is this for? (mobile / dashboard / both)
    3. Who is the audience? (end_user for mobile app users, admin for club/company admins)
    4. Which feature area does this belong to? (list existing areas by running `ls docs/de/` and showing them as options, or let developer type a new one)
    5. What type of article? (guide / faq / troubleshooting / overview)

    **Step 2: Check for Slug Collision**
    Derive slug from the article name. Run:
    ```
    npx tsx src/skill/new-article.ts check-slug "<slug>" --area "<area>"
    ```
    If collision detected (exists: true), warn and ask: "An article already exists at {path}. Overwrite it, or pick a different name?"

    **Step 3: Generate German Draft**
    Run:
    ```
    npx tsx src/skill/new-article.ts generate --feature "<name>" --app "<app>" --audience "<audience>" --area "<area>" --type "<type>"
    ```
    Show the output to the developer. Ask:
    - **approve** -- proceed to write and translate
    - **edit** -- ask what to change, then re-run generate with refined parameters
    - **cancel** -- abort, no files written

    **Step 4: Write and Translate**
    On approval, run:
    ```
    echo '<approved content>' | npx tsx src/skill/new-article.ts write-and-translate --slug "<slug>" --area "<area>"
    ```
    Report which files were written (German original + all translations).

    - Feature area is asked explicitly (not inferred)
    - Slug is auto-generated from the article title using buildSlug() for consistency
    - Manual articles land in the same `docs/{lang}/{area}/` tree as scan-generated articles
    - Manual articles are NOT added to feature-map.json (that is scanner output only)

    Verify the file is valid YAML frontmatter + markdown by reading it back.
